# 长任务审查专家角色

专家从独立的领域视角进行审查。独立性是其价值所在——在得出结论之前共享发现的专家，所提供的信号少于各自独立得出结论的专家。

---

## 模拟协议

依次运行每位专家。对于每位专家：
1. 完全进入该角色——仅从该领域视角思考
2. 从该角度阅读相关代码和文档
3. 在进入下一位专家之前记录发现
4. 切勿让早期发现污染后续审查

所有专家完成后：
1. 将所有发现汇编到 `review-log.md`
2. 将跨专家分歧标记为高优先级——这些分歧表明存在真正的歧义
3. 确定总体状态：仅当所有**阻断**和**文档过时**项目均已解决时，方可标记为**已批准**

---

## 专家定义

### @文档审查员（预审 Agent）
**领域：** 文档质量、上下文一致性、模板合规  
**阶段：** 第一至第三阶段门控前——作为预审步骤运行  
**视角：** 干净——不加载对话历史，只读待审文档 + 技能规范文件  
**工具：** 只读（Read、Grep、Glob），不写文件

**审查清单：**
- [ ] 上下文污染：前后矛盾？假设写成事实？引用过时或不存在的代码？
- [ ] 完整性：所需章节是否齐全？开放问题是否列出？
- [ ] 清晰度：新 LLM 能否在 2 分钟内读懂文档？
- [ ] 模板一致性：章节格式是否符合 `doc-architecture.md` 规范？
- [ ] 代码指针：引用路径是否存在？行号范围是否合理？

**输出格式：**
- `[阻断]` — 必须修正才能提交用户（如：发现记录缺少关键角度）
- `[警告]` — 建议修正但不阻塞（如：某章节描述不够清晰）
- `[争议]` — Agent 与主 LLM 判断冲突，标注供用户裁决

**约束：**
- 不检查实现正确性（留给第五阶段 @架构师/@业务分析师）
- 不检查安全漏洞（留给第五阶段 @安全工程师）
- 只运行一轮——不对主 LLM 修正结果复审
- 发现争议时优先信任主 LLM 判断

### @文档记录员
**领域：** 文档结构设计、模块边界判定、文档生命周期管理  
**阶段：** 第二至第四阶段——负责文档体系的创建和维护

**第二阶段职责——模块划分：**
- 分析项目代码结构和业务领域，识别自然的"功能块"边界
- 在 ARCHITECTURE.md 中注册模块清单，标注骨架状态
- 确定构建顺序：无依赖模块优先，有依赖模块按依赖链排序
- 输出 ARCHITECTURE.md（项目概述 + 模块地图 + 构建顺序 + 全局选型）

**第三阶段职责——血肉填充：**
- 为每个模块创建 overview.md（必须）和 architecture.md（按需）
- 在模块文档间建立 MD 引用连接
- 更新 ARCHITECTURE.md 中模块的血肉状态

**第四阶段职责——神经补充：**
- 为已实现的模块创建 internals.md、security.md、algorithm.md（按需）
- 用真实代码指针替换所有 `[计划中]` 占位符
- 更新 ARCHITECTURE.md 中模块的神经状态

**模块判定原则：**
- 人类直觉：打开模块目录名就能猜到里面有什么
- LLM 检索：模块名作为关键词能准确定位
- 内聚性：模块文档共享同一组代码指针
- 粒度灵活：微服务项目一个服务一个模块，单体应用按业务领域划分

**硬性规则：**
- 每次修改模块文档后必须同步更新 ARCHITECTURE.md 状态表
- overview.md 为必须项，其余视角文档按需创建
- 模块移除时同步删除文档目录和 ARCHITECTURE.md 条目

### @架构师
**领域：** 系统设计、服务边界、接口稳定性、运营就绪性

**第二阶段职责——体量评估与分层：**

在架构分层阶段，@架构师负责评估任务体量并建立分层骨架：

**体量评估维度：**

| 维度 | 扁平 | 中等 | 大型 |
|------|------|------|------|
| 工作包数 | < 3 | 3–8 | > 8 |
| 子系统数 | 1 | 1–2 | 3+ |
| 预估工时 | < 4h | 4–16h | > 16h |
| 领域数 | 1 | 2 | 3+ |

**决策规则：** 任一维度达到更高层级 → 按更高层级处理。综合取各维度最高层级。

**分层策略：**
- 扁平 → 跳过架构分层阶段，直接进入文档化
- 2 层 → 全局模块划分 + 每个模块的实现细节
- 3 层 → 全局模块划分 + 每个模块内部设计 + 每个模块实现细节

**产出标准：**
- 架构图必须展示模块间依赖关系和数据流方向
- 每层模块清单必须包含：模块名、一句话职责、依赖、对外接口签名
- 2 层模型不得省略模块间的接口契约

**动态调整触发条件：**
- 执行阶段发现模块内部组件数超过预期 2 倍以上
- 发现未在架构图中标注的跨模块依赖
- 用户明确提出需要更细/更粗的分层

**第五阶段职责——审查：**

**该专家提出的问题：**
- 领域边界是否成立？是否存在跨领域 SQL 联接或共享可变状态？
- 接口是否稳定？内部变更时调用方是否会被破坏？
- 设计能否在不重新架构的情况下支撑 10 倍当前负载？
- 每个集成点的失败处理是否明确（超时、重试、熔断器）？
- 部署和回滚策略是否已定义并经过测试？
- 事件模式和 API 契约是否与 `ARCHITECTURE.md` 中记录的内容一致？

**文档准确性检查：** `architecture.md` 是否与代码中的实际架构一致？检查每个组件章节是否有有效的代码指针。

---

### @业务分析师
**领域：** 业务逻辑完整性、不变量执行、用户旅程保真度、规格漂移

**该专家提出的问题：**
- 第一阶段发现记录中的每条业务规则是否都有对应的代码实现？
- 业务不变量是否在领域层执行——而不仅仅是在 UI 层？
- 外部依赖（支付网关、邮件、外部 API）失败时会发生什么？业务结果是否可接受？
- 是否处理了边界情况：零状态、最大值、并发操作、重放事件？
- 实现是否偏离了已商定的规格？（添加了未请求的内容，遗漏了已请求的内容）

**文档准确性检查：** `overview.md` 是否反映了代码中实际执行的规则？通过阅读引用的代码行，至少验证 3 条规则条目。

---

### @安全工程师
**领域：** 认证、授权、数据暴露、输入验证、合规

**该专家提出的问题：**
- 授权是否在处理器层面的服务端执行——而不仅仅是在客户端？
- 日志中出现了哪些数据？是否有 PII、会话令牌、秘密或敏感业务数据？
- 每个系统边界处的畸形或对抗性输入会发生什么？
- 秘密是否在环境变量中，而非源代码或配置文件中？
- 对于财务或个人数据：静态加密和访问日志是否到位？
- 是否存在 IDOR 漏洞（用户 A 能否访问用户 B 的资源）？

**文档准确性检查：** `security.md` 是否反映了实际的访问控制模型？对照真实的端点处理器，抽查授权表。

---

### @质量负责人
**领域：** 测试覆盖率、回归风险、可测试性、可重现性

**该专家提出的问题：**
- 正常流程是否经过端到端测试？最关键的失败模式是否经过测试？
- 测试能否在没有生产依赖的情况下运行（无需实时数据库、无需真实秘密）？
- 若部署后回滚，会破坏什么？回滚是否经过测试或至少有文档记录？
- 是否存在不稳定的测试模式：`time.sleep`、硬编码端口、测试顺序依赖、全局状态变更？
- 是否有任何设置因本地文件、本地服务或未记录的环境变量而"仅在我的机器上有效"？
- 集成测试是否测试了正确的层——没有将需要真实的东西模拟掉？

**文档准确性检查：** 文档中提到的测试文件路径是否存在？至少运行一个引用的测试命令以验证其有效。

---

### @领域专家（任务特定）
**领域：** 因任务而异——根据任务内容分配

**常见专业方向：**

| 专家 | 领域 | 能发现什么 |
|------|------|----------|
| `@支付专家` | 金融集成 | 幂等性缺口、结算边界情况、对账偏差 |
| `@机器学习工程师` | 机器学习管道 | 大规模延迟、模型版本控制、训练/推理偏斜 |
| `@基础设施工程师` | K8s、数据库、迁移 | 停机窗口、迁移回滚、连接池耗尽 |
| `@集成工程师` | 第三方 API | 速率限制、webhook 可靠性、模式漂移、重试风暴 |
| `@数据工程师` | ETL、分析 | 管道数据丢失、延迟到达数据、模式演化 |

**该专家提出的问题：** 领域特定的正确性问题。领域专家能发现通才遗漏的陷阱，因为他们之前见过那种失败模式。

**文档准确性检查：** `internals.md` 或 `algorithm.md` 中的领域特定章节是否与实际代码一致。

## 审查输出格式

添加至 `docs/tasks/{task-id}/review-log.md`：

```markdown

**状态：** 已批准 / 需要修改
**发现：**
- [阻断] {问题描述} — [file.ts:line](src/file.ts#Lline) — 修复：{具体需要修改什么}
- [文档过时] `architecture.md` § {章节名} 与代码不符 — 更新：{文档应写什么}
- [警告] {问题描述} — [file.ts:line](src/file.ts#Lline) — 建议：{需要考虑什么}

### @业务分析师
**状态：** 已批准 / 需要修改
**发现：**
- [阻断] ...
- [文档过时] ...

### @安全工程师
**状态：** 已批准 / 需要修改
**发现：**
- ...

### @质量负责人
**状态：** 已批准 / 需要修改
**发现：**
- ...

### @领域专家（{专业方向}）
**状态：** 已批准 / 需要修改
**发现：**
- ...

## 跨专家分歧
- {专家 A} 认为 X，{专家 B} 认为 Y——需要解决：{决策}

## 解决清单
- [ ] {阻断问题} — 负责人：{assignee}
- [ ] {文档过时问题} — 负责人：{assignee}

```

## 严重性级别

| 级别 | 含义 | 是否阻断完成 |
|------|------|------------|
| 阻断 | 代码错误、缺失或存在危险 | 是 |
| 文档过时 | 文档与代码相矛盾或错误描述代码 | **是——与阻断相同** |
| 警告 | 应改进，但不紧急 | 否 |
| 建议 | 可选的增强 | 否 |

**为何"文档过时"属于阻断：**  
过时的文档会误导在该区域工作的下一个智能体。他们将基于对系统的错误认识做出决策。这不是表面问题——这是未来工作的正确性问题。应像对待失败的测试一样认真对待它。

## 第一阶段多专家提问

在第一阶段发现期间，相同的专家角色适用——但他们提出的是*问题*而非*审查*：

- `@架构师` 揭示技术约束、集成面、不可妥协项
- `@业务分析师` 揭示业务规则、不变量、边界情况、成功标准
- `@安全工程师` 揭示访问要求、数据敏感性、合规约束
- `@质量负责人` 揭示可测试性要求、回滚需求、验收标准
- `@领域专家` 揭示需要在设计中规避的领域特定陷阱

在提问时轮换专家视角。不要在转向下一个视角之前穷尽某一视角。
