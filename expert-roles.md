# 长任务审查专家角色

专家从独立的领域视角进行审查。独立性是其价值所在——在得出结论之前共享发现的专家，所提供的信号少于各自独立得出结论的专家。

---

## 模拟协议

依次运行每位专家。对于每位专家：
1. 完全进入该角色——仅从该领域视角思考
2. 从该角度阅读相关代码和文档
3. 在进入下一位专家之前记录发现
4. 切勿让早期发现污染后续审查

所有专家完成后：
1. 将所有发现汇编到 `review-log.md`
2. 将跨专家分歧标记为高优先级——这些分歧表明存在真正的歧义
3. 确定总体状态：仅当所有**阻断**和**文档过时**项目均已解决时，方可标记为**已批准**

---

## 专家定义

### @架构师
**领域：** 系统设计、服务边界、接口稳定性、运营就绪性

**该专家提出的问题：**
- 领域边界是否成立？是否存在跨领域 SQL 联接或共享可变状态？
- 接口是否稳定？内部变更时调用方是否会被破坏？
- 设计能否在不重新架构的情况下支撑 10 倍当前负载？
- 每个集成点的失败处理是否明确（超时、重试、熔断器）？
- 部署和回滚策略是否已定义并经过测试？
- 事件模式和 API 契约是否与 `_INDEX.md` 中记录的内容一致？

**文档准确性检查：** `tech-arch.md` 是否与代码中的实际架构一致？检查每个组件章节是否有有效的代码指针。

---

### @业务分析师
**领域：** 业务逻辑完整性、不变量执行、用户旅程保真度、规格漂移

**该专家提出的问题：**
- 第一阶段发现记录中的每条业务规则是否都有对应的代码实现？
- 业务不变量是否在领域层执行——而不仅仅是在 UI 层？
- 外部依赖（支付网关、邮件、外部 API）失败时会发生什么？业务结果是否可接受？
- 是否处理了边界情况：零状态、最大值、并发操作、重放事件？
- 实现是否偏离了已商定的规格？（添加了未请求的内容，遗漏了已请求的内容）

**文档准确性检查：** `business-logic.md` 是否反映了代码中实际执行的规则？通过阅读引用的代码行，至少验证 3 条规则条目。

---

### @安全工程师
**领域：** 认证、授权、数据暴露、输入验证、合规

**该专家提出的问题：**
- 授权是否在处理器层面的服务端执行——而不仅仅是在客户端？
- 日志中出现了哪些数据？是否有 PII、会话令牌、秘密或敏感业务数据？
- 每个系统边界处的畸形或对抗性输入会发生什么？
- 秘密是否在环境变量中，而非源代码或配置文件中？
- 对于财务或个人数据：静态加密和访问日志是否到位？
- 是否存在 IDOR 漏洞（用户 A 能否访问用户 B 的资源）？

**文档准确性检查：** `security.md` 是否反映了实际的访问控制模型？对照真实的端点处理器，抽查授权表。

---

### @质量负责人
**领域：** 测试覆盖率、回归风险、可测试性、可重现性

**该专家提出的问题：**
- 正常流程是否经过端到端测试？最关键的失败模式是否经过测试？
- 测试能否在没有生产依赖的情况下运行（无需实时数据库、无需真实秘密）？
- 若部署后回滚，会破坏什么？回滚是否经过测试或至少有文档记录？
- 是否存在不稳定的测试模式：`time.sleep`、硬编码端口、测试顺序依赖、全局状态变更？
- 是否有任何设置因本地文件、本地服务或未记录的环境变量而"仅在我的机器上有效"？
- 集成测试是否测试了正确的层——没有将需要真实的东西模拟掉？

**文档准确性检查：** 文档中提到的测试文件路径是否存在？至少运行一个引用的测试命令以验证其有效。

---

### @领域专家（任务特定）
**领域：** 因任务而异——根据任务内容分配

**常见专业方向：**

| 专家 | 领域 | 能发现什么 |
|------|------|----------|
| `@支付专家` | 金融集成 | 幂等性缺口、结算边界情况、对账偏差 |
| `@机器学习工程师` | 机器学习管道 | 大规模延迟、模型版本控制、训练/推理偏斜 |
| `@基础设施工程师` | K8s、数据库、迁移 | 停机窗口、迁移回滚、连接池耗尽 |
| `@集成工程师` | 第三方 API | 速率限制、webhook 可靠性、模式漂移、重试风暴 |
| `@数据工程师` | ETL、分析 | 管道数据丢失、延迟到达数据、模式演化 |

**该专家提出的问题：** 领域特定的正确性问题。领域专家能发现通才遗漏的陷阱，因为他们之前见过那种失败模式。

**文档准确性检查：** `solution-detail.md` 或 `algorithm.md` 中的领域特定章节是否与实际代码一致。

## 审查输出格式

添加至 `docs/tasks/{task-id}/review-log.md`：

```markdown
## 审查会话：{YYYY-MM-DD}

### @架构师
**状态：** 已批准 / 需要修改
**发现：**
- [阻断] {问题描述} — [file.ts:line](src/file.ts#Lline) — 修复：{具体需要修改什么}
- [文档过时] `tech-arch.md` § {章节名} 与代码不符 — 更新：{文档应写什么}
- [警告] {问题描述} — [file.ts:line](src/file.ts#Lline) — 建议：{需要考虑什么}

### @业务分析师
**状态：** 已批准 / 需要修改
**发现：**
- [阻断] ...
- [文档过时] ...

### @安全工程师
**状态：** 已批准 / 需要修改
**发现：**
- ...

### @质量负责人
**状态：** 已批准 / 需要修改
**发现：**
- ...

### @领域专家（{专业方向}）
**状态：** 已批准 / 需要修改
**发现：**
- ...

## 跨专家分歧
- {专家 A} 认为 X，{专家 B} 认为 Y——需要解决：{决策}

## 解决清单
- [ ] {阻断问题} — 负责人：{assignee}
- [ ] {文档过时问题} — 负责人：{assignee}

## 总体状态：待定 / 已批准
```

---

## 严重性级别

| 级别 | 含义 | 是否阻断完成 |
|------|------|------------|
| 阻断 | 代码错误、缺失或存在危险 | 是 |
| 文档过时 | 文档与代码相矛盾或错误描述代码 | **是——与阻断相同** |
| 警告 | 应改进，但不紧急 | 否 |
| 建议 | 可选的增强 | 否 |

**为何"文档过时"属于阻断：**  
过时的文档会误导在该区域工作的下一个智能体。他们将基于对系统的错误认识做出决策。这不是表面问题——这是未来工作的正确性问题。应像对待失败的测试一样认真对待它。

## 第一阶段多专家提问

在第一阶段发现期间，相同的专家角色适用——但他们提出的是*问题*而非*审查*：

- `@架构师` 揭示技术约束、集成面、不可妥协项
- `@业务分析师` 揭示业务规则、不变量、边界情况、成功标准
- `@安全工程师` 揭示访问要求、数据敏感性、合规约束
- `@质量负责人` 揭示可测试性要求、回滚需求、验收标准
- `@领域专家` 揭示需要在设计中规避的领域特定陷阱

在提问时轮换专家视角。不要在转向下一个视角之前穷尽某一视角。
