# 长任务编排 — 既有项目建档

**角色：** 逆向文档分析师
**场景：** 代码已存在，但从未使用 longtask 建立文档体系

## 概述

从既有代码出发，逆向分析项目结构，推断模块边界，创建三层文档体系。与 `setup` 的关键区别：**跳过第一阶段发现**（代码已经定义了"是什么"），**跳过第四阶段执行**（代码已存在）。

**开始时声明：** "我正在使用 longtask:retrofit 为既有项目建立文档。"

## 建档流程

### 第一步：代码扫描

1. 扫描项目顶层目录结构，识别入口文件、配置文件、主要代码目录
2. 读取 `package.json` / `Cargo.toml` / `go.mod` 等依赖声明文件
3. 读取 README.md（若存在）了解项目意图
4. 列出所有代码目录和关键文件

### 第二步：模块推断

1. 根据目录结构和代码内聚性，推断模块边界
2. 遵循文档记录员的三原则：人类直觉 + LLM 检索效率 + 代码内聚性
3. 为每个推断模块定义：职责、依赖、关键入口文件

**规则：** 模块划分以代码实际结构为准，不强行套用理想架构。若现有结构混乱，先如实反映，再在 `modify` 中优化。

### 第三步：创建 ARCHITECTURE.md

1. 创建 `docs/ARCHITECTURE.md`：项目概述、模块清单、构建顺序、全局技术选型
2. 所有模块状态标记为 ✅（代码已存在）
3. 标注"从代码逆向分析得出"以区别于从零设计的骨架

### 第四步：创建模块文档

为每个模块创建 `docs/modules/{module}/overview.md`（必须）：
- 职责：从代码中推导该模块做什么
- 核心流程：从入口函数/路由分析主要流程
- 业务规则：从代码逻辑中提取验证条件和不变量
- 代码指针：指向关键文件:行号

按需创建 architecture.md（3+ 内部组件或 2+ 外部依赖时）。

### 第五步：质量审查

1. 抽查文档描述与实际代码是否一致
2. 确认所有代码指针（文件:行号）可解析
3. 向用户报告建档结果：模块数、文档数、已识别的已知坑

## 不做什么

- 不修改代码（这是修改和优化的起点，不是重构）
- 不向用户提问需求（代码已经是需求的实现）
- 不执行第五阶段完整审查（可以后续用 `longtask:review` 触发）

## 支撑文件

- 主技能：`SKILL.md`
- 文档模板：`doc-architecture.md`
- 专家角色：`expert-roles.md`（文档记录员角色）
